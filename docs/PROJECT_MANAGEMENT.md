# 项目管理机制

> 对抗 AI 遗忘，保证项目持续进行

---

## 核心原则

**外部持久化 > 内部记忆**

- 不依赖 AI 的内部记忆
- 所有重要信息都存储在外部系统
- 每次工作都更新外部状态

---

## 1. 状态快照（PROGRESS.md）

### 作用

- 对抗遗忘的核心文件
- 每次 session 开始时读取
- 每次工作结束时更新

### 内容

```markdown
# 当前进度

## 正在进行的任务
- Issue #1: DATA-001
- 状态: 开发中
- 开始时间: 2026-03-17 23:00
- 当前步骤: 正在实现 update_daily_data 方法
- 已完成步骤:
  - [x] 分析 qlib 数据结构
  - [x] 设计接口
  - [ ] 实现 update_daily_data

## 阻塞问题
- 无 / 问题描述

## 待讨论问题
- 无 / GitHub Discussions 链接
```

### 更新频率

- 开始工作时：读取
- 每完成一个步骤：更新
- 遇到问题时：更新阻塞问题
- 结束工作时：更新

## 2. 心跳机制（Cron Job）

### 作用

- 定期检查项目状态
- 确保项目一直在进行中
- 自动推进停滞的任务

### 配置

```json
{
  "id": "project-heartbeat",
  "schedule": {"kind": "every", "everyMs": 600000}, // 每 10 分钟
  "payload": {
    "kind": "agentTurn",
    "message": "检查项目状态：1. 读取 PROGRESS.md 2. 检查是否有正在进行的任务 3. 检查是否有阻塞问题 4. 如果有阻塞问题，尝试解决或发起讨论 5. 如果没有正在进行的任务，认领下一个任务"
  }
}
```

### 心跳检查逻辑

```
1. 读取 PROGRESS.md
2. 检查"正在进行的任务"
   - 如果有，检查是否超过 1 小时没有更新
     - 如果超过，检查是否遇到问题
   - 如果没有，认领下一个 P0 任务
3. 检查"阻塞问题"
   - 如果有，尝试解决
   - 如果无法解决，在 GitHub Discussions 中发起讨论
4. 汇报状态到群聊
```

## 3. 决策记录（GitHub Discussions）

### 作用

- 记录所有讨论和决策
- 不会丢失
- 可追溯

### 使用场景

- 遇到技术选型问题
- 遇到设计方案问题
- 需要用户决策的问题

### 流程

```
1. 在 GitHub Discussions 中创建讨论
2. 描述问题
3. 提供选项和建议
4. 等待用户或其他机器人回复
5. 记录决策
6. 继续执行
```

---

## 4. 工作流程

### Session 开始时

```
1. 读取 PROGRESS.md
2. 了解当前状态
3. 检查 GitHub Issues
4. 继续上次的工作
```

### 工作中

```
1. 开始新步骤时，更新 PROGRESS.md
2. 完成步骤时，更新 PROGRESS.md
3. 在 Issue 中添加进展评论
4. 遇到问题时，记录到 PROGRESS.md
```

### 遇到问题时

```
1. 分析问题
2. 如果可以自己解决，直接解决
3. 如果需要讨论，在 GitHub Discussions 中创建讨论
4. 等待决策
5. 记录决策
6. 继续执行
```

### Session 结束时

```
1. 更新 PROGRESS.md
2. 在 Issue 中添加进展评论
3. 汇报状态
```

## 5. 质量保证

### 代码质量

- 编码规范（PEP 8）
- Type hints
- Docstrings
- Code review（通过 PR）

### 测试质量

- 单元测试
- 测试覆盖率 > 80%
- 自动化测试（pytest）

### 文档质量

- 设计文档
- API 文档
- 使用示例

## 6. 协作机制

### main（PM）职责

- 设计和规划
- 决策
- 协调
- 质量把控

### mac（开发者）职责

- 编码实现
- 编写测试
- 修复 Bug
- 更新文档

### 协作流程

```
1. main 分配任务（通过 GitHub Issue）
2. mac 认领任务并开始开发
3. mac 遇到问题时：
   a. 分析问题
   b. 如果需要讨论，通知 main
   c. main 在 GitHub Discussions 中创建讨论
   d. 共同讨论并做出决策
   e. mac 继续执行
4. mac 完成任务，创建 PR
5. main 审查 PR
6. 合并代码
```

---

## 7. 应急机制

### 如果 mac 长时间无响应

1. main 检查 mac 的状态
2. 尝试通过 sessions_send 联系
3. 如果仍然无响应，通知用户

### 如果遇到无法解决的问题

1. 在 GitHub Discussions 中记录问题
2. 通知用户
3. 等待用户决策

---

## 8. 持续改进

### 定期回顾

- 每周回顾项目进展
- 分析遇到的问题
- 改进工作流程

### 文档更新

- 更新设计文档
- 更新任务分解
- 更新进度追踪

---

**这个机制的核心是：不依赖 AI 的内部记忆，而是依赖外部持久化系统（GitHub）。**
